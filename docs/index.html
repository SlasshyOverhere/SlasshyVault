SlasshyVault is a local-first desktop application. Every component is designed so you control your data at all times.

🔐 Security & Privacy Evidence

Every claim below is backed by evidence you can verify yourself in the source code.

✅ No Telemetry or Analytics

The entire codebase has zero calls to analytics services. Grep for yourself:

git grep -i "google-analytics\|segment\.io\|amplitude\|mixpanel\|sentry\|datadog\|newrelic" src/ src-tauri/
# → No results. Zero analytics dependencies.

There is no `window.analytics`, no tracking pixel, no crash reporter. Sentry is listed in Cargo.toml but is not initialized anywhere in the app — it's a leftover unused dependency.

✅ No Network Requests on Launch

The app makes zero HTTP requests when it starts. No "phone home," no license check, no version ping. Verified by tracing all fetch/invoke calls:

# All network requests go through explicit user actions:
- User clicks "Scan" → GDrive API
- User searches → TMDB API
- User plays → GDrive stream URL
- User signs in → OAuth flow

✅ OAuth Tokens Never Leave Your Machine

The OAuth flow is a standard Google OAuth 2.0 authorization_code grant:

src-tauri/src/gdrive.rs:
// Token stored in GoogleTokens struct, serialized to local config file
// Used only for Bearer Authorization headers to api.googleapis.com
// NEVER transmitted anywhere else

The refresh token is stored locally in `media_config.json` (obfuscated). Only the app reads it. The backend never sees your tokens — it only sees the one-time authorization code from Google, exchanges it, and passes the result back through a session ID.

✅ Watch Together Relay Is Your Own

The relay is a Cloudflare Worker deployed to your Cloudflare account using an API token you provide. The worker source is bundled into the app binary at src-tauri/relay/worker.js and uploaded directly to Cloudflare's API — no middleman.

You can verify exactly what runs on your account by reading the 200-line worker file. It uses Durable Objects for ephemeral room state — no database, no persistence.

✅ Backend Is Stateless — No Database

The OAuth backend has no database, no Redis, no file storage. State is kept in-memory with a 5-minute TTL. After the OAuth flow completes, the session is deleted. The entire backend is ~120 lines of Express:

SlasshyVault-Backend/server.js:
- /auth/google → redirect
- /auth/callback → exchange code → store in Map → redirect
- /auth/session/:id → return tokens → delete from Map
- /auth/refresh → forward to Google
- /health → plain OK

✅ No User Accounts, No Database

There is no user registration, no email/password flow, no user database. "Sign in" just means "grant my app access to my Google Drive." The app identifies you by your Google ID only to the extent that Drive API needs it — and that ID is never sent anywhere but Google.

🖥️ Self-Hosting the OAuth Backend

The only external service SlasshyVault requires is a tiny OAuth backend to handle Google's code exchange. You can host it yourself in 5 minutes.

Architecture

The flow is simple:

  1. User clicks "Sign in" in the app
  2. App opens browser to https://your-backend/auth/google?nonce=...
  3. Backend redirects to Google's consent screen
  4. Google redirects to https://your-backend/auth/callback?code=xxx&state=yyy
  5. Backend exchanges the code for tokens
  6. Backend stores tokens in memory with a session ID
  7. Backend redirects browser to http://localhost:8085/callback?session_id=zzz
  8. App picks up the session, fetches tokens from backend, stores them locally
  9. Done. Backend never hears from the app again until next sign-in.

Deploy to Render (Free)

1. Fork/clone the repo
2. Go to https://dashboard.render.com → New Web Service
3. Connect your GitHub repo
4. Settings:
   - Root directory: SlasshyVault-Backend
   - Build command: npm install
   - Start command: npm start
5. Add environment variables:
   GOOGLE_CLIENT_ID=...
   GOOGLE_CLIENT_SECRET=...
   REDIRECT_URI=https://your-app.onrender.com/auth/callback
6. Deploy
7. In Google Cloud Console → APIs & Services → Credentials
   → Your OAuth client → Add Authorized redirect URI:
   https://your-app.onrender.com/auth/callback

Deploy to Any VPS

git clone https://github.com/SlasshyOverhere/SlasshyVault
cd SlasshyVault/SlasshyVault-Backend
npm install

# Create .env:
echo 'GOOGLE_CLIENT_ID=xxx
GOOGLE_CLIENT_SECRET=xxx
REDIRECT_URI=https://your-domain.com/auth/callback
PORT=3001
OAUTH_CALLBACK_URL=http://localhost:8085/callback' > .env

# Run with PM2 or systemd:
node server.js

Environment Variables

VariableRequiredDescription
GOOGLE_CLIENT_ID✅From Google Cloud Console
GOOGLE_CLIENT_SECRET✅From Google Cloud Console
REDIRECT_URI✅Must match Google's authorized redirect URIs. Format: https://your-domain/auth/callback
OAUTH_CALLBACK_URL—Where to redirect after auth. Default: http://localhost:8085/callback. Only change if you modified the Tauri OAuth listener port.
PORT—Server port (default 3001)

Setting the URL in the App

The Tauri app reads the backend URL from gdrive.rs line 18:

const AUTH_SERVER_URL: &str = "https://slasshyvault.onrender.com";

To point it at your own backend, you have two options:

  1. Env var at build time — set STREAMVAULT_AUTH_SERVER_URL=https://your-backend.com
  2. Modify the source — change the constant and rebuild

There is no runtime config for the auth URL (by design — prevents phishing). You must build your own copy.

❓ FAQ

Does SlasshyVault collect any data?

No. There are no analytics, telemetry, crash reports, or usage tracking of any kind. The app makes no network requests on launch. All data lives on your machine and in your Google Drive.

What does the OAuth backend do with my data?

Nothing. It stores a session ID in memory for 5 minutes max, then deletes it. It has no database, no file storage, and no logging of user data. Full source is at SlasshyVault-Backend/server.js — ~120 lines.

Does the Watch Together relay send data to any server?

No. The relay deploys to your own Cloudflare account. The source code is bundled in the app at src-tauri/relay/worker.js. It uses Durable Objects (ephemeral in-memory state) — no database, no persistence, no central server.

What third-party APIs does the app call?

TMDB (api.themoviedb.org) for metadata, and optionally imdbapi.dev or OMDb for ratings. You can provide your own API keys in Settings. These calls are direct from your machine — no proxy involved.

Can I use SlasshyVault without Google Drive?

No. Google sign-in is required because the app uses Google Drive as its cloud storage backend. All media files you access through the cloud must be in your Drive. Local files still work, but you need to sign in to use the app.

Is there a mobile version?

No. SlasshyVault is a desktop app built with Tauri. There are currently no mobile builds.

🔏 Privacy Policy

Last updated: June 30, 2026

1. What We Collect

Nothing. SlasshyVault does not collect, store, transmit, or share any personal data. There are no analytics, no telemetry, no crash reporters, no advertising identifiers, no fingerprinting, and no user tracking of any kind. This is a verifiable fact — the full source code is available for audit.

2. Google Drive Access

When you sign in with Google, the Software requests the following OAuth scopes:

  • https://www.googleapis.com/auth/drive — Read and stream your media files
  • https://www.googleapis.com/auth/userinfo.email — Display your email in settings

This access is used exclusively for:

  • Listing folders and files you choose to index
  • Streaming video content directly to your media player

The Software never writes, modifies, or deletes files in your Drive without your explicit action. Your OAuth tokens are stored locally on your machine (obfuscated in media_config.json) and are never sent to any third party. You can revoke access at any time from your Google Account permissions page.

3. OAuth Backend Server

The OAuth backend server is a stateless Node.js service that handles only the Google OAuth code exchange. It stores OAuth state and session IDs in memory for a maximum of 5 minutes, after which they are permanently deleted. The backend has:

  • No database (no Turso, no SQLite, no Postgres)
  • No Redis (no Upstash, no key-value store)
  • No persistent storage of any kind
  • No logging of user information
  • No analytics

The backend cannot identify you, cannot track you, and retains nothing after your OAuth flow completes. Full source is at SlasshyVault-Backend/server.js (~120 lines).

4. Watch Together Relay

The Watch Together feature uses a Cloudflare Worker deployed to your own Cloudflare account. This Worker relays WebSocket messages between participants in a Watch Together room using Durable Objects (ephemeral in-memory state). It stores no data beyond the active room session. State is lost when the last participant disconnects and after 6 hours of inactivity. You can delete the Worker at any time from your Cloudflare dashboard or through the Software settings.

5. Third-Party API Calls

The Software may make direct HTTPS requests to the following services:

  • TMDB (api.themoviedb.org) — Movie/TV metadata, posters, search. Request includes your search query or media ID.
  • imdbapi.dev (api.imdbapi.dev) — IMDb ratings, episode data. Request includes the IMDb ID.
  • OMDb (www.omdbapi.com) — Optional alternative for IMDb ratings, only if you provide a key.
  • Google APIs (www.googleapis.com, oauth2.googleapis.com) — Drive file listing, streaming, OAuth.
  • Cloudflare API (api.cloudflare.com) — Only when you use the one-click relay deploy feature, using the API token you provide.

These requests include only the data necessary for the operation (search query, media ID, file ID) and never include personal information. These services operate under their own privacy policies.

6. Data Storage

  • Configuration — Stored in media_config.json in the app data directory. Includes your API keys (if provided), relay URL, Cloudflare token (if provided), and preferences.
  • Watch History — Stored in a local SQLite database. Includes what you watched, when, and progress. Never uploaded anywhere.
  • OAuth Tokens — Stored obfuscated in media_config.json, used only for Google Drive API calls.
  • Image Cache — Downloaded posters and stills cached locally for performance.

7. Data Deletion

  • Local data: Use "Factory Reset" in Settings to delete all local data (watch history, config, cache).
  • OAuth tokens: Revoke at Google Account permissions.
  • Cloudflare Worker: Delete from app Settings ("Stop Relay") or Cloudflare dashboard.
  • Google Drive files: Nothing is written to your Drive by the app. No cleanup needed.

The Software is not intended for children under 13. No data is collected from any user regardless of age.
